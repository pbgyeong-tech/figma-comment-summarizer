# Git Commit Convention

이 프로젝트는 AI와 개발자가 히스토리를 쉽게 파악할 수 있도록 **Conventional Commits** 규칙을 따릅니다.
모든 커밋 메시지는 다음 형식을 준수해야 합니다.

## 1. 기본 구조 (Structure)

```text
<type>(<scope>): <subject>

<body>
- Header: 필수. 변경 사항의 종류와 요약을 적습니다.
- Body: 선택. 변경한 이유와 상세 내용을 적습니다.

2. Type (종류)
- feat: 새로운 기능 추가
- fix: 버그 수정
- style: 코드 스타일 변경 (포맷팅, 세미콜론 등)
- docs: 문서 수정
- test: 테스트 코드 추가 또는 수정
- refactor: 코드 리팩토링
- perf: 성능 개선
- chore: 빌드 시스템, 패키지 매니저 등 설정 파일 변경

3. 범위
변경사항이 영하받은 파일이나 모듈을 명시합니다.
- ui (HTML/CSS 관련)
- api (서버 통신 관련)
- manifest (설정 파일 관련)
- utils (유틸리티 함수)

4. 메시지 작성 규칙
4-1. 제목(Subject):
- 명령문으로 작성합니다. (예: "수정했음" -> "수정")
- 50자를 넘기지 않도록 노력합니다.
- 끝에 마침표(.)를 찍지 않습니다.
- 언어 선택: 한글로 통일합니다.
4-2. 본문(Body):
- "무엇을"보다 "왜" 변경했는지에 집중해서 작성합니다.
- 이전 코드와 비교하여 달라진 점을 AI가 추적하기 쉽도록 구체적으로 적습니다.

5. 예시
5-1. 새로운 기능 추가 시
feat(ui): 코멘트 요약 버튼 추가
사용자가 클릭하면 API를 호출할 수 있도록 메인 화면 하단에 '요약하기' 버튼을 추가함.
버튼 스타일은 피그마 디자인 시스템을 따름.
5-2. 버그 수정 시
fix(auth): API 토큰 저장 안 되는 문제 수정
manifest.json 권한 설정 누락으로 인해 setAsync가 동작하지 않던 문제 해결.
storage 권한을 추가함.
